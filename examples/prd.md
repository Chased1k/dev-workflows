# The Perfect Cut — Product Requirements Document

> Generated by PRD Builder grill-me session. This is the contract for implementation.

## Overview

The Perfect Cut is a competition weight management app for combat athletes (BJJ, MMA, wrestling, boxing, Muay Thai). It helps athletes answer four questions: Can I realistically make this weight? What should I do today? Am I on track? What is the safest strategy for my weigh-in format?

It is NOT a calorie counter, weight loss app, or generic nutrition tracker. It is a performance cut planner with an AI coach that reads structured data.

## Primary User

Combat sport athletes (18-45) who compete in weight-class sports. They need to cut weight safely while maintaining performance. Currently using spreadsheets, coach advice, or trial-and-error. They weigh in at competitions with formats ranging from same-day to night-before.

## Core Features (MVP)

### Feature 1: Cut Feasibility Assessment

- **User story:** As an athlete, I want to know if I can realistically make my target weight so that I don't attempt an unsafe cut.
- **Acceptance criteria:**
  - [ ] Computes feasibility score (1-10) from current weight, target weight, days remaining, weigh-in format
  - [ ] Computes difficulty score (1-10) independently
  - [ ] Shows risk flags for aggressive cuts
  - [ ] Rules engine computes scores — never the LLM
- **States:**
  - Loading: skeleton placeholder while engine computes
  - Empty: prompt to set up event first
  - Error: "Unable to calculate — check your event setup"
  - Success: score card with breakdown, risk flags, recommendation
  - Edge cases: same-day weigh-in with large cut (>5% bodyweight), very long timeline (>12 weeks)
- **Inputs:** current weight, target weight class, event date, weigh-in format
- **Outputs:** feasibility score, difficulty score, risk flags, recommendation text

### Feature 2: Daily Dashboard

- **User story:** As an athlete, I want to see what I should do today at a glance so that I stay on track.
- **Acceptance criteria:**
  - [ ] Shows daily targets: calories, water (L), sodium (mg), fiber (g), weight target
  - [ ] Shows today's log status (completed vs. pending)
  - [ ] Shows trajectory: on track / ahead / behind
  - [ ] Fight week mode triggers stricter display 7 days before weigh-in
- **States:**
  - Loading: skeleton cards
  - Empty: "Set up your event to get daily targets"
  - Error: "Unable to load targets"
  - Success: target cards with progress indicators
  - Edge cases: no event configured, event in the past, multiple active events

### Feature 3: Daily Logging

- **User story:** As an athlete, I want to log my daily metrics so the AI coach can give me accurate feedback.
- **Acceptance criteria:**
  - [ ] Log weight, food intake, water intake, sodium, fiber, sleep hours, training minutes, symptoms
  - [ ] Validates inputs (no negative values, reasonable ranges)
  - [ ] Saves to weight_entries and daily_logs tables
  - [ ] Shows confirmation with comparison to targets
- **States:**
  - Loading: form skeleton
  - Empty: blank form with today's date
  - Error: validation errors per field, save failure
  - Success: "Logged! You're [on track / ahead / behind]"
  - Edge cases: multiple entries same day (update, not duplicate), missing optional fields

### Feature 4: AI Coach Chat

- **User story:** As an athlete, I want to talk to an AI coach that knows my data so I get personalized advice.
- **Acceptance criteria:**
  - [ ] Builds context packet from DB before every AI turn
  - [ ] Context includes: athlete profile, active event, weigh-in format, last 14 days of logs, current targets, risk flags
  - [ ] AI explains targets — never invents numerical targets
  - [ ] Routes cheap model for logging chores, smart model for coaching
  - [ ] Symptoms/questions/risk flags always route smart
  - [ ] Generates structured summary after each session, saves to ai_context_snapshots
- **States:**
  - Loading: "Building your context..."
  - Empty: welcome message with suggested questions
  - Error: "Coach is unavailable — check your connection"
  - Success: streaming response with disclaimer
  - Edge cases: no event configured (coach explains what's needed), no recent logs (coach notes limited data)

### Feature 5: Rehydration Protocol

- **User story:** As an athlete, I want a post-weigh-in rehydration plan so I recover safely for competition.
- **Acceptance criteria:**
  - [ ] Generates protocol based on weigh-in format and cut magnitude
  - [ ] Includes fluid volume, electrolyte targets, carbohydrate targets, timeline
  - [ ] Rules engine generates — not LLM
  - [ ] Conservative defaults, warns on aggressive protocols
- **States:**
  - Loading: computing protocol
  - Empty: "Complete your event setup to generate a protocol"
  - Error: "Unable to generate — check event details"
  - Success: phased protocol with timeline
  - Edge cases: same-day weigh-in (compressed timeline), very small cut (minimal protocol)

## Non-Goals (MVP)

- Coach dashboard (multi-athlete management)
- Payments / subscriptions
- Wearables integration
- Barcode scanning for food
- Photo-based body fat estimation
- Social features
- Metric/imperial unit preference toggle (deferred to Phase 1)

## Data Model

### Entity: athlete_profiles

| Field | Type | Required | Notes |
|---|---|---|---|
| id | uuid | yes | PK, linked to auth.users |
| current_weight_kg | numeric | yes | Updated on each weigh-in |
| height_cm | numeric | no | For body composition context |
| role | text | yes | athlete, coach, or admin |
| organization_id | uuid | no | Phase 2 readiness |
| coach_id | uuid | no | Phase 2 readiness |

### Entity: events

| Field | Type | Required | Notes |
|---|---|---|---|
| id | uuid | yes | PK |
| athlete_id | uuid | yes | FK → athlete_profiles |
| target_weight_kg | numeric | yes | Target weight class |
| weigh_in_date | timestamptz | yes | When they step on the scale |
| weigh_in_format | text | yes | same-day, night-before, morning-of, 30-min-before |
| organization_id | uuid | no | Phase 2 readiness |

### Entity: daily_plans

| Field | Type | Required | Notes |
|---|---|---|---|
| id | uuid | yes | PK |
| event_id | uuid | yes | FK → events |
| date | date | yes | Which day this plan is for |
| calorie_target | integer | yes | Computed by rules engine |
| water_target_ml | integer | yes | Computed by rules engine |
| sodium_target_mg | integer | yes | Computed by rules engine |
| fiber_target_g | integer | yes | Computed by rules engine |
| weight_target_kg | numeric | yes | Expected weight this day |

### Entity: weight_entries

| Field | Type | Required | Notes |
|---|---|---|---|
| id | uuid | yes | PK |
| athlete_id | uuid | yes | FK → athlete_profiles |
| weight_kg | numeric | yes | Measured weight |
| measured_at | timestamptz | yes | When measured |

### Entity: daily_logs

| Field | Type | Required | Notes |
|---|---|---|---|
| id | uuid | yes | PK |
| athlete_id | uuid | yes | FK → athlete_profiles |
| date | date | yes | Which day |
| calories_consumed | integer | no | |
| water_consumed_ml | integer | no | |
| sodium_consumed_mg | integer | no | |
| fiber_consumed_g | integer | no | |
| sleep_hours | numeric | no | |
| training_minutes | integer | no | |
| symptoms | text[] | no | Array of symptom tags |

### Relationships

- athlete_profiles → events: one-to-many (one athlete, many events)
- events → daily_plans: one-to-many (one event, many daily plans)
- athlete_profiles → weight_entries: one-to-many
- athlete_profiles → daily_logs: one-to-many

## Technical Constraints

- **Stack:** Next.js 14 (App Router), Supabase (Postgres + Auth), Tailwind CSS + shadcn/ui, AI SDK v6 via Vercel AI Gateway
- **Platform:** Web (mobile-first), future Expo mobile app
- **Performance:** Dashboard loads <2s, AI responses stream within 3s
- **Security:** RLS on all tables, Supabase Auth, API key for AI gateway only
- **Integrations:** Supabase Auth, Vercel AI Gateway

## Design Direction

- **Visual tone:** Clean, professional, combat-sport aesthetic — dark mode with accent colors for status (green=on track, yellow=ahead/behind, red=risk)
- **Platform priority:** Mobile-first — used on phones at gyms and during fight week
- **Design system:** shadcn/ui components, Tailwind CSS
- **Key screens:**
  1. Dashboard — today's targets, log status, trajectory
  2. Log — daily entry form
  3. Assess — feasibility score card
  4. Coach — AI chat interface
  5. Calendar — daily plan view
  6. Settings — profile + event setup
  7. Rehydration — post-weigh-in protocol

## Open Questions

- Metric/imperial unit preference: user-requested, partial support exists but not yet wired to athlete profile or UI
